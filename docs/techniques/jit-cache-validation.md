# JIT Code Cache Structural Validation

> Validates the structural integrity of ART's JIT Code Cache memory regions to detect injected trampolines or code hidden within JIT-managed memory.

---

## Overview

ART's JIT compiler maintains a Code Cache region in memory mapped from `/memfd:jit-cache`. This region has a well-defined structure: one RW data segment and one RX code segment per cache instance, with consistent inodes across related mappings. Attackers can inject trampoline code into JIT Code Cache free space to hide execution in "legitimate" JIT memory, bypassing anonymous executable page detection. Structural validation of the JIT Code Cache layout, inode consistency, and segment count reveals such injections.

---

## Injection Side

### How Attackers Use This Technique

1. **Locate JIT Code Cache** — Scan `/proc/self/maps` for `/memfd:jit-cache` entries to identify the base address and size of each JIT cache segment.
2. **Find free space** — Locate unused space within the JIT cache's RW region (data view) where the allocator has not placed JIT-compiled code.
3. **Write trampoline code** — Copy attacker-crafted trampoline (e.g., `LDR Xn, #literal; BR Xn`) into the free space.
4. **Adjust permissions** — Call `mprotect()` on the page to set RX if it is not already executable, or rely on the existing RX code view that maps the same underlying memfd.
5. **Patch target function** — Overwrite the target function's entry point to branch into the JIT cache region.
6. **Result** — The branch target resides within the JIT cache address range, passing naive "is target in JIT region?" whitelisting checks.

**Alternative (more sophisticated):**

1. **Hook `JitCodeCache::CommitCode()`** — Intercept ART's internal JIT code commit function.
2. **Piggyback on legitimate allocation** — Insert attacker code alongside or in place of a legitimate JIT compilation result.
3. **Appears as normal JIT output** — The injected code occupies a properly allocated JIT code slot, making it structurally indistinguishable from real JIT output without deeper semantic analysis.

### Artifacts

| Artifact                  | Location                       | Indicator                                                     |
| ------------------------- | ------------------------------ | ------------------------------------------------------------- |
| Extra executable segments | `/proc/self/maps`              | More than 1 `r-xp` segment for `jit-cache`                    |
| Inode inconsistency       | `/proc/self/maps` vs `fstat()` | Different inodes for related jit-cache mappings               |
| Unexpected segment count  | `/proc/self/maps`              | Total jit-cache segments >4 (normal: rw, rx, and possibly ro) |
| Code in data region       | JIT cache rw segment           | Executable instructions found in RW data area                 |
| Trampoline patterns       | JIT cache rx region            | `LDR`+`BR` or `B` patterns not generated by ART JIT compiler  |
| Address sequence anomaly  | JIT cache mappings             | Non-contiguous or overlapping address ranges                  |

### Evasion Techniques

| Evasion                         | Description                                                                                 | Bypass Difficulty |
| ------------------------------- | ------------------------------------------------------------------------------------------- | ----------------- |
| Perfect structural compliance   | Match exact segment layout of legitimate JIT cache — requires precise mimicry of all fields | ★★★★★             |
| Hook `JitCodeCache::CommitCode` | Inject via JIT's own allocation path — appears as legitimate compiler output                | ★★★★★             |
| Minimal injection               | Write very small trampolines that blend with JIT code style and alignment                   | ★★★★              |
| ART version adaptation          | Adapt to version-specific JIT cache layout changes across Android releases                  | ★★★★              |

## Detection Side

### Mechanism

A clean ART JIT Code Cache has specific structural invariants:

1. All `jit-cache` and `jit-zygote-cache` mappings from the same memfd share the same inode.
2. There should be exactly 1 RX (executable) segment per JIT cache instance.
3. The dual-view (RW data view + RX code view) maps to the same underlying memfd with consistent inode.
4. Total segments per cache ≤ 4 (typically: RW data, RX code, plus possibly RO).
5. Address ranges should be contiguous within each cache instance.

Any violation of these invariants indicates memory injection.

### Anti-Evasion Properties

| Property                    | Explanation                                                                                                            |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| Resistant to libc hooks     | `/proc/self/maps` and `fstat` invoked via SVC directly                                                                 |
| Resistant to GOT/PLT hijack | Custom maps parsing with no library calls                                                                              |
| SVC bypass benefit          | Maps data comes from kernel — accurate regardless of userspace hooks                                                   |
| Remaining bypass surface    | Attacker hooking `JitCodeCache::CommitCode` to inject via the legitimate path is extremely hard to detect structurally |

### Detection Strategy

1. **Read `/proc/self/maps` via SVC** — Use `openat` + `read` syscalls directly (ARM64: SVC #0) to obtain the memory map without libc hooks.
2. **Extract JIT cache entries** — Parse all lines containing `jit-cache` or `jit-zygote-cache`.
3. **Validate per cache type:**
   - a. Count executable segments (permission contains `x`) — must be exactly 1.
   - b. Collect all inodes — must all be identical for the same cache name.
   - c. Count total segments — should not exceed 4.
4. **Cross-validate inodes via `fstat()`** — For each `jit-cache` entry, call `fstat()` on the corresponding fd via `/proc/self/fd/` using SVC and compare the inode with the value from maps.
5. **Detect fake mappings** — Inode mismatch between maps and fstat indicates a fabricated jit-cache mapping (injection).
6. **Verify address contiguity** — Parse address ranges for each cache instance and verify they are contiguous (no unexpected gaps or interleaved entries from other mappings).
7. **Optional deep scan** — Read RX region content and scan for non-JIT trampoline patterns. `LDR Xn, #literal → BR Xn` at method entry boundaries is not emitted by ART's JIT compiler and indicates inline hook trampolines.

### Detection PoC

```c
// JIT Code Cache Structural Validation
// Detect injected code within JIT cache by validating structural invariants

#include <sys/stat.h>
#include <linux/fcntl.h>

#define SYS_openat_ARM64   56
#define SYS_read_ARM64     63
#define SYS_fstat_ARM64    80
#define SYS_close_ARM64    57

// SVC wrappers (ARM64)
static long svc_openat(int dirfd, const char *path, int flags) {
    register long x8 __asm__("x8") = SYS_openat_ARM64;
    register long x0 __asm__("x0") = dirfd;
    register long x1 __asm__("x1") = (long)path;
    register long x2 __asm__("x2") = flags;
    __asm__ volatile("svc #0" : "=r"(x0) : "r"(x8), "r"(x0), "r"(x1), "r"(x2));
    return x0;
}

static long svc_fstat(int fd, struct stat *buf) {
    register long x8 __asm__("x8") = SYS_fstat_ARM64;
    register long x0 __asm__("x0") = fd;
    register long x1 __asm__("x1") = (long)buf;
    __asm__ volatile("svc #0" : "=r"(x0) : "r"(x8), "r"(x0), "r"(x1));
    return x0;
}

typedef struct {
    unsigned long start;
    unsigned long end;
    char perms[5];
    unsigned long inode;
    char name[256];
} MapEntry;

int validate_jit_cache(void) {
    int result = 0;

    // Step 1: Read /proc/self/maps via SVC
    int fd = svc_openat(AT_FDCWD, "/proc/self/maps", O_RDONLY);
    if (fd < 0) return -1;
    // ... read and parse maps content ...

    // Step 2: Extract jit-cache entries
    MapEntry jit_entries[16];
    int jit_count = 0;
    // parse_maps_for(buf, "jit-cache", jit_entries, &jit_count);

    // Step 3a: Count executable segments — must be exactly 1
    int rx_count = 0;
    for (int i = 0; i < jit_count; i++) {
        if (jit_entries[i].perms[2] == 'x') {
            rx_count++;
        }
    }
    if (rx_count != 1) {
        result |= FLAG_JIT_EXTRA_RX_SEGMENT;  // Injection detected
    }

    // Step 3b: Validate inode consistency
    unsigned long expected_inode = jit_entries[0].inode;
    for (int i = 1; i < jit_count; i++) {
        if (jit_entries[i].inode != expected_inode) {
            result |= FLAG_JIT_INODE_MISMATCH;  // Fake mapping detected
        }
    }

    // Step 3c: Validate segment count
    if (jit_count > 4) {
        result |= FLAG_JIT_EXCESS_SEGMENTS;  // Unexpected layout
    }

    // Step 4: Cross-validate via fstat on /proc/self/fd/<N>
    // Extract fd number from maps path, then fstat and compare inode
    struct stat st;
    char fd_path[64];
    // snprintf(fd_path, sizeof(fd_path), "/proc/self/fd/%d", memfd_num);
    int memfd = svc_openat(AT_FDCWD, fd_path, O_RDONLY);
    if (memfd >= 0) {
        svc_fstat(memfd, &st);
        if ((unsigned long)st.st_ino != expected_inode) {
            result |= FLAG_JIT_FSTAT_INODE_MISMATCH;
        }
        // close via SVC
    }

    // Step 6: Verify address contiguity
    for (int i = 1; i < jit_count; i++) {
        if (jit_entries[i].start != jit_entries[i - 1].end) {
            // Check if gap is occupied by a non-jit-cache entry
            result |= FLAG_JIT_ADDRESS_GAP;
        }
    }

    return result;
}
```

### False Positive Risks

| Scenario                                   | Mitigation                                                                                                               |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| Zygote-inherited `jit-zygote-cache`        | Separate validation for `jit-cache` vs `jit-zygote-cache`; each has its own invariants and expected segment layout       |
| JIT cache compaction/GC                    | JIT GC does not change segment count — it only reclaims content within existing segments                                 |
| Multi-process JIT sharing (Android 14+)    | Newer ART versions may use shared JIT cache; track per-Android-version baselines for expected segment counts             |
| Profile-guided compilation changing layout | AOT profile compilation does not affect JIT cache structure — it is a separate concern from JIT code cache memory layout |
